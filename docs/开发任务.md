## 第一阶段目标

**验收标准**：`go run ./cmd/server` 起服务 → 用 `wscat` 带 JWT 连上 → 发 `{"op":"ping"}` → 收到 `{"op":"pong"}`。**不涉及 Kafka、不涉及 DB 消息表**，纯协议骨架。

### 要动的文件

```
internal/
├── ws/                      🆕
│   ├── frame.go            op 枚举 + Frame struct + 各 payload 类型
│   ├── hub.go              uid → *Session 映射 + register/unregister/push
│   ├── session.go          单连接的 readPump / writePump / 心跳
│   └── handler.go          /ws 握手：Upgrade + 鉴权 + 注册到 Hub
├── router/router.go        把 /ws 接到 ws.HandleWS
└── cmd/server/main.go      启动 hub.Run() goroutine
```

**还不碰的**：
- `internal/mq/` — 第二阶段加 Kafka
- `internal/worker/` — 第三阶段加消费者
- `messages` 表 — 等 Kafka 链路通了再加

### 实现优先级（一个 commit 一小步）

1. `frame.go` — 把上次设计的 op / Frame / SendData 等 struct 落成代码
2. `hub.go` — Hub + Session 的骨架，带 register/unregister channel
3. `session.go` — readPump 循环读帧、writePump 循环发帧、`SetReadDeadline` 心跳
4. `handler.go` — 走 `middleware.JWTAuth()` → Upgrade → `hub.Register(uid, sess)`
5. 接到 `router.Init`，改造 `main.go` 把 `hub` 注进去
6. 测试：`ping/pong` 通，`send` 暂时只回一个假 `ackok`（msg_id 写死字符串）

---

## 第二阶段目标（第一阶段通了之后）

接入 Kafka 的**上行链路**（send → upstream → persist → DB）。只做单聊，不做扇出推送。这一阶段完成后：
- 客户端发消息 → 消息落 MySQL `messages` 表
- 但对方**还收不到**（没接 downstream 推送）
- 客户端可以通过 HTTP `/conv/:id/history` 拉历史看到

这样能把写路径和读路径的复杂度拆开，一个一个调。

---

## 第三阶段（扇出 + 推送）

加 `fanout-worker` + `chat.downstream.{gw_id}` + `pusher-worker`——这一步通了，对方就能实时收到消息。此时**核心 IM 闭环完成**，后面加群聊、好友、拉黑都是业务逻辑叠加。

**现在直接开始第一阶段吗？** 我会：

1. 先读一下你现有的 `config.go` 和 `middleware/auth.go` 确认接口形状
2. 按顺序写 `frame.go` → `hub.go` → `session.go` → `handler.go`
3. 改 `router.go` 和 `main.go`
4. 最后给你一条 `wscat` 测试命令

确认就干，或者你有什么想先调整的（比如想先把 `Login/Refresh/Logout` 三个 HTTP handler 补完再动 WS）都可以。
