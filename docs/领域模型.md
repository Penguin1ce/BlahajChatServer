## 1. 领域模型

```
users          (uid, email, password_hash, nickname, avatar)
friendships    (uid_a, uid_b, status, created_at)         # uid_a < uid_b 去重
friend_apply   (id, from_uid, to_uid, status, reason, ts) # pending/accepted/rejected
blocks         (uid, blocked_uid, created_at)             # 单向拉黑
groups         (gid, owner_uid, name, avatar, created_at)
group_members  (gid, uid, role, muted_until, joined_at)   # role: owner/admin/member
group_apply    (id, gid, applicant_uid, status, reason, ts)
conversations  (conv_id, type, peer_key)                  # type: c2c/group
user_conv      (uid, conv_id, last_read_msg_id, unread, pinned, muted)
messages       (msg_id, conv_id, from_uid, type, content, created_at, deleted_at)
```

**conv_id 的生成**（很关键，决定 Kafka 分区均衡）：
- 单聊：`conv_id = hash("c2c:" + min(uidA,uidB) + ":" + max(uidA,uidB))`
- 群聊：`conv_id = "g:" + gid`

**msg_id 用 Snowflake**（64bit：时间戳 + workerID + seq）——全局唯一、趋势递增、可直接当游标（比 UUID 好太多，面试官爱问为什么不用 UUID）。

## 2. Kafka 设计

### 2.1 为什么用 Kafka？

面试时这样答：
- **削峰**：大群 500 人同时在线，一条消息要扇出 500 次推送，同步做会把 Gateway 打挂
- **解耦**：发消息的关键路径只需要"落 Kafka"，后续的落库、扇出、推送、审计、搜索索引都是异步消费者组
- **可回溯**：Kafka 日志本身就是消息流，消费者挂了能 rewind，比纯 Redis pub/sub 强

### 2.2 Topic 规划

| Topic                     | 分区 Key  | 用途                                   | 消费者           |
| ------------------------- | --------- | -------------------------------------- | ---------------- |
| `chat.upstream`           | `conv_id` | 客户端所有上行消息入口                 | persist / fanout |
| `chat.downstream.{gw_id}` | `uid`     | 每个 Gateway 一个，只收自己要推的      | 对应 Gateway     |
| `chat.notify`             | `uid`     | 好友/入群申请、审批结果等系统消息      | notify-consumer  |
| `chat.event`              | `conv_id` | 已读回执、撤回、输入中等轻量事件       | fanout           |
| `chat.dlq`                | —         | 死信（格式错、权限校验失败的消息留痕） | 人工 / 告警      |

### 2.3 关键决策：分区 Key = conv_id

**为什么不用 uid？**
- Kafka **只保证分区内有序**。用 `conv_id` 做 key，同一会话的所有消息永远进同一个分区，天然保证"A 和 B 的对话里消息严格有序"。
- 如果用 uid，A 给 B 发两条，第二条因为重分区可能比第一条先落库，出现乱序。

### 2.4 消费者组

```
chat.upstream
    ├─ persist-group   (N 实例)  → 写 MySQL messages 表
    ├─ fanout-group    (M 实例)  → 查成员 → 写 chat.downstream.{gw_id}
    └─ audit-group     (可选)    → ES / 风控
```

每个组独立位点，互不影响。**这就是用 Kafka 不用 Redis Stream 的理由**——多消费者组、位点隔离、吞吐高。

### 2.5 生产者配置（细节分）

```go
acks=all                  // 至少一副本确认
enable.idempotence=true   // 幂等生产（同一 producer 重试不重复）
max.in.flight=5
retries=Int.Max
compression=lz4
linger.ms=5               // 小批聚合，提吞吐
```

## 3. Client ↔ Server 协议

### 3.1 登录（接上一问）
- `POST /auth/login` → `{access, refresh, expires_in}`
- 短命 access（15min）+ 长命 refresh（30d），你现在的代码已经对

### 3.2 WebSocket 握手
```
GET /ws?token=<access>
Sec-WebSocket-Protocol: blahaj.v1
```
在 `middleware.JWTAuth()` 里校验，Upgrade 后从 `c.Get(consts.CtxUserID)` 取 uid。

### 3.3 帧格式（二进制更省，文本调试方便，建议 JSON 起步）

```jsonc
// 上行
{ "op": "send", "seq": 42, "data": {
    "client_msg_id": "uuid",     // 幂等去重
    "conv_id": "c2c:abc",
    "type": "text|image|file",
    "content": "hello"
}}
{ "op": "ack",  "seq": 43, "data": { "msg_id": 12345 } }  // 客户端收到下行后 ACK
{ "op": "read", "data": { "conv_id": "...", "msg_id": 12345 } }
{ "op": "ping" }

// 下行
{ "op": "msg",    "data": { 完整 message } }
{ "op": "ackok",  "seq": 42, "data": { "msg_id": 12345, "ts": 17... } }
{ "op": "notify", "data": { "type": "friend_apply", ... } }
{ "op": "pong" }
```

`seq` 是客户端自增号，服务端 `ackok` 带回，用于客户端匹配"哪条发送成功了"。

## 4. 写路径（发一条消息发生了什么）

```
Client ──send──▶ Gateway (WS)
                    │ 1. 校验 token、解出 uid
                    │ 2. 查权限：conv_id 类型 → 
                    │     c2c:   对方是否拉黑了我 (blocks Redis 缓存)
                    │     group: 我是否在群、是否禁言
                    │ 3. Redis SETNX client_msg_id (5min) 做前置幂等
                    │ 4. 生成 snowflake msg_id
                    │ 5. Produce chat.upstream (key=conv_id)
                    │ 6. 立即回 ackok {msg_id}  ← 对客户端来说消息已"发送成功"
                    ▼
              Kafka chat.upstream
                    │
       ┌────────────┼─────────────┐
       ▼            ▼             ▼
   persist       fanout        audit
   (MySQL)    │                (ES)
              │ 查 conv 成员列表（Redis 缓存）
              │ 对每个在线成员：查路由表 Redis: user:gw:{uid} = gw_id
              │ 按 gw_id 聚合 → Produce chat.downstream.{gw_id} (key=uid)
              ▼
         Kafka chat.downstream.gw-3
              ▼
         Gateway-3 消费 → 查本地 uid→conn → WS 推送
              ▼
         Client 收到 → 回 ack
```

**几个面试会追问的点**：

- **为什么 Gateway 先回 ackok，不等落库？** → 为了 p99 延迟。落库异步，只要 Kafka acks=all 成功就算"不会丢"。面试官问数据一致性，回答："Kafka 是 Source of Truth，MySQL 是物化视图"。
- **Gateway 宕机、消息还没 Produce 成功？** → 客户端超时重试（带相同 `client_msg_id`），新 Gateway 会因 Redis 幂等键或 DB 唯一索引兜底。
- **怎么防刷屏？** → Redis 令牌桶，key = `rate:{uid}`，每秒 N 条。

## 5. 读路径 & 扇出（大群的关键权衡）

### 5.1 写扩散 vs 读扩散

|          | 写扩散（每个收件人一份） | 读扩散（只存一份，按 conv 读）       |
| -------- | ------------------------ | ------------------------------------ |
| 单聊     | ✅ 简单，2 份而已         | 也行                                 |
| 500 人群 | ❌ 写 500 次，存储爆炸    | ✅ 只写 1 份                          |
| 未读数   | 容易（count inbox）      | 要 `user_conv.last_read_msg_id` 对比 |
| 多端同步 | 麻烦                     | ✅ 天然一致                           |

**推荐：统一读扩散**。
- `messages` 表存一份
- `user_conv` 存每个用户在每个会话的 `last_read_msg_id`
- 未读数 = `messages.max(msg_id) > last_read_msg_id` 的 count（Redis 实时维护增量即可，不用每次查 DB）

### 5.2 在线路由表

```
Redis: HSET user:route {uid} {gw_id}       # 用户上线时 Gateway 写入
       SADD gw:users:{gw_id} {uid}         # 反向索引，便于 Gateway 优雅下线
       EXPIRE 2 * heartbeat_interval       # 心跳续期
```

- Gateway 启动时 subscribe 自己的 `chat.downstream.{gw_id}`
- 优雅下线：停止接收新连接 → 把自己的用户踢下线（推通知让客户端重连到别的 Gateway）→ commit 位点 → 退出
- 多端登录：`user:route:{uid}` 改成 SET，一个用户多个 gw_id（手机+网页同时在线）

### 5.3 离线消息

- 用户离线时 fanout 发现路由表查不到 → 不推 WS，但消息已经落库
- 用户上线后发 `GET /sync?conv_id=xxx&after_msg_id=xxx` 增量拉
- 或者登录后走 `/conv/list` 拉会话列表 + 未读数，点开会话再拉消息

## 6. 好友 / 群 / 拉黑 / 入群申请

都是"写 DB + 发一条 `chat.notify`"的套路：

**好友申请**
```
POST /friend/apply  {to_uid, reason}
  → INSERT friend_apply
  → Produce chat.notify (key=to_uid, type=friend_apply)
  → fanout 查路由表 → 推 WS
POST /friend/apply/:id/accept
  → UPDATE status, INSERT friendships (双向)
  → Produce chat.notify → 对方收到
```

**入群申请**：同上，多一步检查 group 的加入模式（自由/审批/禁止）。

**拉黑**：
- `POST /block/:uid` → INSERT blocks + 删 Redis 缓存
- 写路径中 Gateway 校验 blocks，**静默丢弃**（返回 ackok 但不送达，别让对方探测到自己被拉黑了——这是产品细节）
- 同时应该把现有好友关系删掉/标记

## 7. 可靠性 & 一致性

### 幂等（三层防线）
1. **Producer 幂等**（Kafka 原生）
2. **Redis `SETNX client_msg_id`**（5min TTL，挡住客户端重试）
3. **DB UNIQUE(from_uid, client_msg_id)** 兜底

### 消息不丢
- 客户端 → Gateway：`client_msg_id` + 重试
- Gateway → Kafka：`acks=all` + 重试
- Consumer → DB：**先写 DB 再 commit 位点**（at-least-once，配合幂等 = exactly-once 效果）
- Gateway → Client：推送后等 ACK，超时重推 N 次仍失败则放弃（DB 已存，客户端下次上线拉）

### 顺序
- 分区 key = conv_id 保证同会话有序
- 客户端本地按 `msg_id` 插入（snowflake 趋势递增，乱序到达也能正确排序）

## 8. 简历加分点（怎么讲这个项目）

写简历时重点突出这几条（每条都是面试会深挖的）：

1. **基于 Kafka 的消息分发架构**：用 conv_id 作为分区 key 保证会话级顺序；设计了 4 个 topic + 3 个消费者组，将持久化、扇出、审计解耦；producer 开启幂等 + acks=all，消费者 at-least-once + DB 唯一索引实现端到端 exactly-once。

2. **Gateway 水平扩展**：基于 Redis 的在线路由表 + per-gateway downstream topic，Gateway 无状态可任意扩缩容；支持多端同时在线；优雅下线流程保证不丢消息。

3. **读扩散消息模型**：单聊/群聊统一读扩散，用 snowflake msg_id 做游标，`user_conv.last_read_msg_id` 维护已读指针；大群消息存储从 O(N) 降到 O(1)。

4. **三层幂等防线**：解决 WS 网络抖动下客户端重试导致的消息重复问题（client_msg_id → Redis SETNX → DB 唯一索引）。

5. **JWT + Refresh Token 轮换**：access 无状态校验 + refresh 轮换 + jti 黑名单实现主动注销；WS 握手复用同一套中间件。

6. **可观测性**（可选但很加分）：Prometheus 埋点（在线连接数、消息 QPS、端到端 p99 延迟、Kafka lag）；pprof + zlog trace_id 串联整条链路。
