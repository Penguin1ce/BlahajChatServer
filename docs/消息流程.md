可以把整个发消息流程想成一条流水线：

```text
客户端发消息
→ 服务端检查并落库
→ 当前连接先收到 ackok
→ 消息事件写入 Kafka
→ 每个服务实例从 Kafka 消费
→ 各自投递给自己机器上的 WebSocket 连接
```

核心入口在 [internal/ws/client.go](/Users/firefly/Developer/code/go/BlahajChatServer/internal/ws/client.go:124)。

客户端先发一帧 WebSocket：

```json
{
  "op": "send",
  "seq": 1,
  "data": {
    "client_msg_id": "本地临时ID",
    "conv_id": "会话ID",
    "type": "text",
    "content": {"text": "你好"}
  }
}
```

服务端收到后，`Client.dispatch` 看到 `op = send`，就调用：

```go
service.HandleSend(ctx, c.userID, d)
```

这里是真正的“发送消息业务逻辑”。它会做几件事：

1. 检查参数是否合法，比如 `client_msg_id`、`conv_id`、`type`、`content` 是否存在。
2. 用 Redis 做幂等：同一个用户、同一个 `client_msg_id` 重发，不会重复落库。
3. 检查发送者是不是这个会话的成员。
4. 开 MySQL 事务：
   - 写入 `messages` 表
   - 更新会话最后一条消息
   - 给除发送者外的成员未读数 `+1`

如果这些都成功，说明消息已经真正进数据库了。

然后服务端会先给“当前发送连接”回一个 `ackok`：

```text
ackok = 你的发送请求已经被服务端成功处理并落库
```

这个 `ackok` 只发给当前连接，不代表别人已经收到了消息。客户端可以用它把本地“发送中”的临时消息，替换成服务端正式的 `msg_id`。

如果这是重复发送，也就是 Redis 幂等命中了旧消息，那么流程到这里就结束：只补一个 `ackok`，不会再次广播 `msg`。

如果是新消息，服务端继续查这个会话有哪些成员：

```go
members, err := dao.ListMembers(ctx, d.ConvID)
```

然后把消息包装成一条 `OpMsg` 帧，再发布到 bus：

```go
bus.Global.Publish(ctx, bus.ChatEvent{
    MsgID:   msg.MsgID,
    ConvID:  d.ConvID,
    Targets: members,
    Frame:   data,
})
```

现在项目只走 Kafka，所以这里实际进入 [internal/bus/kafka.go](/Users/firefly/Developer/code/go/BlahajChatServer/internal/bus/kafka.go:150)。

`KafkaBus.Publish` 会把 `ChatEvent` 序列化成 JSON，然后写入 Kafka。Kafka message 的 key 是：

```go
Key: []byte(e.ConvID)
```

意思是：同一个会话的消息尽量进入同一个 partition，这样 Kafka 可以保证这个会话内的消息顺序。

之后，每个服务实例都会有自己的 Kafka consumer 在跑。consumer 拉到事件后，会调用启动时注入的 `fanout` 回调：

```go
ws.GlobalHub.Broadcast(...)
```

这个 `Hub` 只管本机上的 WebSocket 连接。比如：

```text
instance-1 上有用户 A 的连接
instance-2 上有用户 B 的连接
```

A 发消息后：

```text
instance-1 写 Kafka
→ instance-1 消费到，发现本机有 A，投递给 A
→ instance-2 也消费到，发现本机有 B，投递给 B
```

最后进入 [internal/ws/hub.go](/Users/firefly/Developer/code/go/BlahajChatServer/internal/ws/hub.go:53)：

```text
Hub.Broadcast
→ Hub.Run
→ 对每个目标 uid 调 SendToUser
→ 找到这个 uid 的所有在线连接
→ 把消息塞进每个连接的 send channel
→ Client.writePump 写回 WebSocket
```

所以最终用户收到的是：

```text
op = msg
data = MsgData
```

还有两个很重要的点：

第一，**离线用户不会实时收到 WebSocket 推送**。但数据库已经落库，未读数也已经加了。用户重新上线后，可以通过历史消息接口和会话列表补回来。

第二，发送者自己也会收到一份 `msg`。因为 `Targets` 包含会话所有成员，包括发送者。客户端需要用 `msg_id` 或 `client_msg_id` 做本地去重，避免界面上显示两条一样的消息。

一句话总结：

**`HandleSend` 负责把消息安全地写进数据库；`KafkaBus` 负责把“新消息事件”扩散到所有服务实例；`Hub` 负责把事件送到本机在线 WebSocket 连接。**
